# тЭЧ Golden Rules

## 1. Never mix responsibilities

тЭМ Bad:

* Firestore inside UI
* Logic inside widgets
* Models doing network calls

тЬЕ Good:

* UI тЖТ presentation
* Logic тЖТ application
* Models тЖТ domain
* Firebase тЖТ data

---

## 2. Keep models pure

Models must:

* be immutable
* have no side effects
* not depend on UI or Firebase

---

## 3. Providers control state

* all updates go through notifiers
* UI never mutates state directly

---

## 4. Features are isolated

Each feature:

* owns its logic
* owns its UI
* owns its data

# ЁЯзй Architectural Justification

The chosen structure is based on the following principles:

### Separation of Concerns
Each layer (domain, application, presentation, data) has a clearly defined responsibility, reducing coupling between components.

### Feature-Based Organization
Grouping files by feature (e.g., gameplay, auth) improves scalability and simplifies navigation.

### State Management Isolation
Using Riverpod Notifiers ensures that application state is managed centrally and predictably.

### Testability
The separation between domain models and application logic enables easier unit testing.

### Scalability
New features can be added without affecting existing modules due to clear boundaries.
