# 🏗️ High-Level Architecture

The project is divided into **4 main layers**:

```
main.dart → app/ → core/ → features/
```

---

## 🔹 Layer Overview

| Layer       | Responsibility                    |
| ----------- | --------------------------------- |
| `main.dart` | App entry point                   |
| `app/`      | App shell (routing + root widget) |
| `core/`     | Global shared utilities           |
| `features/` | Actual app functionality          |

---

# 🚀 Entry Point

## `main.dart`

### Responsibility

* Initialize Flutter
* Initialize Firebase
* Wrap app with Riverpod (`ProviderScope`)
* Launch the app

### Rule

❗ Must NOT contain:

* business logic
* UI screens
* Firebase logic
* test code

---

## `app/app.dart`

* Router integration

## `app/router.dart`

# ❗ Golden Rules

## 1. Never mix responsibilities

❌ Bad:

* Firestore inside UI
* Logic inside widgets
* Models doing network calls

✅ Good:

* UI → presentation
* Logic → application
* Models → domain
* Firebase → data

---

## 2. Keep models pure

Models must:

* be immutable
* have no side effects
* not depend on UI or Firebase

---

## 3. Providers control state

* all updates go through notifiers
* UI never mutates state directly

---

## 4. Features are isolated

Each feature:

* owns its logic
* owns its UI
* owns its data

---

# 🧠 Mental Model Summary

When adding a file, ask:

### 1. Is it app startup?

→ `main.dart` or `app/`

### 2. Is it shared globally?

→ `core/` or `shared/`

### 3. Is it feature-specific?

→ `features/<feature>/`

Then:

| Question            | Folder       |
| ------------------- | ------------ |
| What is it?         | domain       |
| How does it change? | application  |
| What does user see? | presentation |
| Where is data from? | data         |

# 🧩 Architectural Justification

The chosen structure is based on the following principles:

### Separation of Concerns
Each layer (domain, application, presentation, data) has a clearly defined responsibility, reducing coupling between components.

### Feature-Based Organization
Grouping files by feature (e.g., gameplay, auth) improves scalability and simplifies navigation.

### State Management Isolation
Using Riverpod Notifiers ensures that application state is managed centrally and predictably.

### Testability
The separation between domain models and application logic enables easier unit testing.

### Scalability
New features can be added without affecting existing modules due to clear boundaries.
